# CPU 文档解析私有化服务 — 技术报告

## 一、最终方案效果

### 1.1 方案概述

自组装全链路文档解析 Pipeline，不依赖 PaddleOCR 端到端包，全部使用 OpenVINO 加速：

```
输入 (图片/扫描件PDF)
  │
  ├─→ 版面检测: PP-DocLayoutV3 (OpenVINO) → 25类区域 + 阅读顺序
  │
  ├─→ 全页OCR: PP-OCRv6 Small (RapidOCR/OpenVINO) → 文字行 + 坐标
  │
  ├─→ 坐标分配: OCR行按中心点匹配到版面区域
  │
  ├─→ 表格解析: 坐标聚类 → Markdown 表格 (无额外模型)
  │
  └─→ Markdown 组装: 按阅读顺序输出结构化内容
```

### 1.2 性能数据 (实测)

| 阶段 | 耗时 | 占比 |
|------|------|------|
| 版面检测 (PP-DocLayoutV3, OpenVINO) | 437ms | 38% |
| 全页OCR (PP-OCRv6 Small, OpenVINO) | 702ms | 62% |
| 坐标分配 + 表格解析 + Markdown组装 | <1ms | ~0% |
| **总计** | **1,139ms/页** | - |

**速度：0.88 pages/s（沙箱受限约1核）**

### 1.3 优化历程

| 版本 | 方案 | 耗时 | 加速比 |
|------|------|------|--------|
| v0 | 裸 Paddle 逐区域推理 | 10,380ms | 1.0x (基线) |
| v1 | OpenVINO版面 + ONNX Runtime全页OCR | 2,890ms | 3.6x |
| **v2 (最终)** | **OpenVINO 全加速** | **1,139ms** | **9.1x** |

## 二、OpenVINO 的架构兼容性分析

### 2.1 OpenVINO 不只是 Intel

常见误解是 OpenVINO 只能跑 Intel CPU。实际情况：

| 架构 | 支持状态 | 性能水平 | 说明 |
|------|---------|---------|------|
| Intel x86 (6代-14代 Core) | 完全支持 | **最优** | AVX2/AVX-512/AMX 深度优化 |
| Intel Xeon (1代-5代) | 完全支持 | **最优** | 服务器首选 |
| AMD x86_64 | 支持 (通用x86路径) | 中等 | 无 AMD 专属优化，走 SSE/AVX2 通用路径 |
| ARM (armv7a+) | 支持 (2023.1+) | 中等 | 有 ARM CPU 插件，持续优化中 |
| ARM64 (arm64-v8a+) | 支持 | 中等偏好 | Apple M1/M2、树莓派4、ARM服务器 |
| Apple Silicon | 支持 | 中等偏好 | macOS ARM64 |
| RISC-V | 预览支持 | 基础 | 生态尚不成熟 |

### 2.2 关键结论

**OpenVINO 在 Intel CPU 上有压倒性性能优势**（利用 MKL-DNN/OneDNN、AVX-512、AMX 等 Intel 专属指令集），但在 AMD 和 ARM 上也能跑，只是性能会打折扣：

- Intel Xeon/Core (AVX-512)：100% 性能（我们的基准）
- AMD EPYC/Ryzen (AVX2)：约 60-80% 性能（无 AVX-512 优化路径）
- ARM64 (Neon/SVE)：约 50-70% 性能（ARM 插件优化程度较低）

**所以如果客户机器架构多样，不能只依赖 OpenVINO 一个后端。**

## 三、不同架构的加速框架选型

### 3.1 总览表

| 客户机器架构 | 推荐加速框架 | 备选方案 | 备注 |
|-------------|-------------|---------|------|
| Intel Xeon/Core (服务器) | **OpenVINO** | ONNX Runtime + OneDNN | 最优解，深度优化 |
| AMD EPYC/Ryzen (服务器) | **ONNX Runtime** | OpenVINO (通用路径) / ZenDNN | ONNX RT 对 AMD 优化更好 |
| ARM64 服务器 (鲲鹏/飞腾/Ampere) | **ONNX Runtime + ACL** | OpenVINO ARM / TFLite | ACL = ARM Compute Library |
| Apple Silicon (M1/M2/M3) | **ONNX Runtime + CoreML** | OpenVINO ARM64 | CoreML 利用 ANE 神经引擎 |
| NVIDIA GPU (有显卡) | **ONNX Runtime + CUDA** / TensorRT | PaddleOCR GPU | 速度最快 |
| AMD GPU (Radeon/Instinct) | **ONNX Runtime + ROCm** | DirectML (Windows) | ROCm 生态在追赶 |
| 国产NPU (华为昇腾/寒武纪) | **MindSpore Lite / CANN** | 厂商SDK | 需单独适配 |
| 海光/兆芯 (x86兼容) | **ONNX Runtime** | OpenVINO (可能兼容) | 本质是 x86，但缺 AVX-512 |
| 低功耗边缘 (RK3588等) | **TFLite / ONNX Runtime** | NCNN / MNN | 有 NPU 可走 RKNN |

### 3.2 各框架详细说明

#### OpenVINO (Intel 首选)
- 开发商：Intel
- 支持架构：x86_64 (Intel 最优)、ARM64、RISC-V
- 加速指令：SSE4.2 → AVX2 → AVX-512 → AMX (逐代增强)
- 优势：Intel CPU 上性能最强，模型自动优化，支持 INT8 量化
- 劣势：AMD 上无专属优化，ARM 优化程度不如原生框架
- 适用：Intel Xeon 服务器、Intel Core 工作站
- 安装：`pip install openvino`

#### ONNX Runtime (最通用)
- 开发商：Microsoft
- 支持架构：x86_64 (Intel/AMD)、ARM64、ARM32
- 加速后端 (Execution Providers)：
  - CPU：默认 (MLAS 内核，支持 x86/ARM)
  - OneDNN：Intel CPU 加速 (可选)
  - ACL：ARM CPU 加速 (ARM Compute Library)
  - CUDA/TensorRT：NVIDIA GPU
  - ROCm：AMD GPU
  - CoreML：Apple 神经引擎
  - OpenVINO：Intel 硬件 (作为 EP 嵌入)
  - DirectML：Windows DirectX 12 GPU
  - QNN：高通 NPU
- 优势：**跨平台最通用**，一套代码适配所有硬件
- 劣势：单一平台性能略逊于该平台专属框架
- 适用：需要一套部署包覆盖多种架构的场景
- 安装：`pip install onnxruntime` (CPU) / `pip install onnxruntime-gpu` (CUDA)

#### AMD ZenDNN
- 开发商：AMD
- 支持架构：AMD Zen 系列 CPU (EPYC, Ryzen)
- 加速方式：针对 Zen 微架构优化的 DNN 原语库
- 优势：AMD CPU 上比通用 ONNX Runtime 快 20-40%
- 劣势：仅支持 AMD，生态较小
- 适用：AMD EPYC 服务器
- 集成：作为 ONNX Runtime 的底层 BLAS 替换

#### ARM Compute Library (ACL)
- 开发商：ARM
- 支持架构：ARM Cortex-A 系列 (ARMv7/ARMv8)
- 加速指令：NEON、SVE、SVE2
- 优势：ARM 原生优化，性能最佳
- 劣势：仅 ARM，需编译集成
- 适用：鲲鹏920、飞腾、Ampere Altra、树莓派
- 集成：ONNX Runtime 编译时开启 `--use_acl`

#### TFLite / NCNN / MNN (边缘端)
- 适用：ARM 嵌入式、手机、低功耗设备
- 优势：极小内存占用，支持 NPU 加速
- 劣势：模型兼容性不如 ONNX
- 适用场景：边缘盒子、手机端

#### TensorRT (NVIDIA GPU)
- 开发商：NVIDIA
- 支持：NVIDIA GPU (Compute Capability 7.0+)
- 优势：GPU 上性能最强，支持 FP16/INT8
- 适用：有 NVIDIA 显卡的服务器
- 速度：比 CPU 快 5-20 倍

### 3.3 推荐策略：分层适配

```
优先级 1 (覆盖80%客户): ONNX Runtime 通用版
  └─ x86_64 (Intel + AMD + 海光 + 兆芯) + ARM64 全覆盖
  └─ 一套包，零配置

优先级 2 (Intel 客户性能最大化): OpenVINO 加速版
  └─ 检测到 Intel CPU → 自动切换 OpenVINO 后端
  └─ 性能提升 2-3 倍

优先级 3 (有 GPU 的客户): CUDA/TensorRT 版
  └─ 检测到 NVIDIA GPU → 切换 CUDA EP
  └─ 性能提升 5-20 倍

优先级 4 (特殊架构): 定制适配
  └─ 华为昇腾 → CANN/MindSpore
  └─ 高通 → QNN/SNPE
  └─ 其他 NPU → 厂商 SDK
```

## 四、多架构部署服务包设计

### 4.1 服务包架构

```
doc-parse-service/
├── app/
│   ├── main.py              # API 服务入口 (FastAPI)
│   ├── parser.py            # 解析核心逻辑 (架构无关)
│   ├── backend_selector.py  # 自动检测硬件 → 选择最优后端
│   └── config.yaml          # 服务配置
├── models/                   # ONNX 模型文件 (通用格式)
│   ├── PP-DocLayoutV3.onnx
│   ├── PP-OCRv6_det_small.onnx
│   └── PP-OCRv6_rec_small.onnx
├── backends/
│   ├── openvino_backend.py  # Intel 优化路径
│   ├── onnxrt_backend.py    # 通用路径 (x86/ARM)
│   └── cuda_backend.py      # GPU 路径 (可选)
├── deploy/
│   ├── Dockerfile.x86       # x86_64 镜像
│   ├── Dockerfile.arm64     # ARM64 镜像
│   ├── install.sh           # 一键安装脚本
│   └── verify.py            # 部署验证脚本
├── requirements.txt
└── README.md
```

### 4.2 硬件自动检测 + 后端选择逻辑

```python
# backend_selector.py 核心逻辑
import platform, subprocess

def detect_backend():
    machine = platform.machine()  # x86_64 / aarch64 / arm64
    
    # 1. 检测 GPU
    if has_nvidia_gpu():
        return "cuda"  # ONNX Runtime + CUDA EP
    
    # 2. 检测 CPU 架构
    if machine == "x86_64":
        vendor = get_cpu_vendor()  # GenuineIntel / AuthenticAMD
        if vendor == "GenuineIntel" and has_avx512():
            return "openvino"  # Intel 最优路径
        else:
            return "onnxrt"    # AMD / 海光 / 兆芯 → ONNX Runtime
    
    elif machine in ("aarch64", "arm64"):
        return "onnxrt_acl"    # ARM → ONNX Runtime + ACL
    
    else:
        return "onnxrt"        # 兜底
```

### 4.3 模型格式统一

所有后端统一使用 **ONNX 格式** 模型（.onnx），这是关键设计决策：

| 后端 | 模型加载方式 |
|------|-------------|
| OpenVINO | 直接读 .onnx → 自动转 IR → 推理 |
| ONNX Runtime | 直接读 .onnx → 推理 |
| TensorRT | .onnx → trtexec 转 .engine → 推理 |
| TFLite | .onnx → onnx2tf → .tflite (需预转) |

**好处：一套模型文件适配所有后端，无需为每个架构单独转换。**

## 五、部署前准备与验证清单

### 5.1 客户环境信息收集

部署前需向客户确认：

| 项目 | 需要确认的内容 | 影响 |
|------|--------------|------|
| CPU 架构 | x86_64 / aarch64 / 其他 | 决定安装包和后端 |
| CPU 型号 | Intel Xeon Gold 6348? AMD EPYC 7763? 鲲鹏920? | 决定加速策略 |
| CPU 指令集 | AVX2? AVX-512? AMX? NEON? SVE? | 影响推理性能 |
| CPU 核数 | 物理核 / 可用核 | 决定 Worker 数 |
| 内存 | 总量 / 可用 | 模型加载 + 并发 |
| GPU | 有/无，型号，显存 | 是否走 GPU 加速 |
| 操作系统 | Ubuntu 22.04? CentOS 7? Windows? | 依赖兼容性 |
| 内核版本 | uname -r | 某些特性需要新内核 |
| 容器环境 | 是否 Docker/K8s | 影响资源限制和驱动挂载 |
| 网络 | 是否离线 | 决定是否需要离线安装包 |
| Python 版本 | 3.8+ | 依赖兼容性 |

### 5.2 部署验证脚本 (verify.py)

每次部署后自动运行，验证环境是否满足要求：

```python
验证项目:
1. CPU 架构检测 → 确认匹配安装包
2. 指令集检测 (AVX2/AVX-512/NEON) → 确认加速可用
3. 内存检测 → 确认 > 2GB 可用
4. 模型文件完整性 → MD5 校验
5. 推理后端加载 → OpenVINO/ONNX Runtime 初始化成功
6. 单页推理测试 → 输出正确 + 耗时在预期范围
7. 并发压力测试 → 3 Worker 同时跑不 OOM
8. 长时间稳定性 → 跑 100 页无内存泄漏
```

## 六、总结与建议

### 6.1 核心结论

1. **OpenVINO 不是只能跑 Intel**，但在 Intel 上性能最优。AMD/ARM 上建议用 ONNX Runtime。
2. **ONNX 模型格式是跨架构的关键**——一套模型文件，多个推理后端。
3. **推荐"ONNX Runtime 通用 + OpenVINO Intel 加速"双后端策略**，覆盖 95% 客户。
4. 有 GPU 的客户额外提供 CUDA 路径，性能提升 5-20 倍。
5. 国产芯片（海光/兆芯）本质是 x86，ONNX Runtime 可直接跑；鲲鹏/飞腾是 ARM64，也有对应方案。

### 6.2 最小可行部署矩阵

| 客户类型 | 占比(估) | 方案 | 额外工作 |
|---------|---------|------|---------|
| Intel 服务器 | 60% | OpenVINO | 零 (当前方案) |
| AMD 服务器 | 15% | ONNX Runtime | 切换后端配置 |
| ARM 服务器 | 10% | ONNX Runtime (ARM) | 换 ARM 包 |
| 有 GPU | 10% | ONNX Runtime + CUDA | 装 CUDA 版 |
| 其他/国产 | 5% | 按需适配 | 单独评估 |
